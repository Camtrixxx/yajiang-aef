# 实验总结：v0.3-c Ascend 4 卡训练与部署导出

## 1. 阶段目标

v0.3-c 的目标是在华为 Ascend 服务器上完成一次可复现的 4 卡训练，并把训练结果整理成可继续训练和可部署的两类模型文件。

本阶段重点验证：

* 代码从 NVIDIA CUDA 环境迁移到 Ascend NPU 环境；
* 新增 `jrc_water` 作为第三个监督目标；
* 全量 `1708` 个 patch 的 manifest 能同时包含 `dem / worldcover / jrc_water`；
* 训练脚本能在后四张 NPU `4,5,6,7` 上通过 DDP 并行运行；
* checkpoint 不再每个 epoch 都保存，避免输出目录膨胀；
* 训练结束后导出一个更轻量、适合部署加载的模型文件；
* 新增 baseline 评测入口，用于后续比较 AEF embedding 与传统特征。

值域处理：


转换命令：

```bash

```

## 7. 如何使用部署模型

部署模型用于把一个 patch 的多源时序输入编码成 AEF embedding。

最常用输出：

```text
embedding_map: [B, embedding_dim, H/2, W/2]
embedding:     [B, embedding_dim]
```

当前 v0.3-c 配置中：

```text
embedding_dim = 128
image_size = 128
embedding_map spatial size = 64 x 64
```

加载示例：

```python
import torch

from src.eval.features import load_deploy_model, extract_aef_embedding_map
from src.utils.device import resolve_device

device = resolve_device("npu:0")
model, cfg = load_deploy_model(
    "outputs/aef_hyh_yajiang_v0_3_c/exports/aef_hyh_yajiang_v0_3_c_deploy.pt",
    device=device,
)

# batch 需要符合 YajiangAEFDataset / aef_collate_fn 的输出协议
embedding_map = extract_aef_embedding_map(model, batch, device)
```

部署模型的典型用途：

* 给每个 patch 生成 128 维全局 embedding；
* 给每个 patch 生成 64 x 64 的 dense embedding map；
* 在少样本下游任务中冻结 AEF，只训练轻量分类/回归头；
* 作为变化检测、地物分类、水体识别等任务的 backbone 特征。

```text

```

支持目标：


示例：WorldCover 多分类评测

```bash

  --target worldcover \
  --device npu:0 \
```

## 9. 当前结论

v0.3-c 是当前第一个完整的 Ascend 4 卡训练版本。

它相比 v0.3-a 的关键进展是：

* 从单卡训练扩展到后四张 Ascend NPU 的 DDP 训练；
* 新增 `jrc_water` 目标，形成 `dem + worldcover + jrc_water` 的三目标重建训练；
* 训练输出从“每 epoch 保存”改为只保留 `best/latest/final`；
* 训练结束自动导出部署模型；
* 初步建立了 embedding 下游 baseline 评测代码，为后续验证 AEF 表征能力打基础。

后续建议：

* 为 baseline 评测建立固定 train/test split，避免每次随机划分导致结果不可比；
* 针对 `worldcover` 类别不均衡，增加 per-class IoU / macro F1 报告；
* 针对 `jrc_water` 稀疏问题，重点报告 `jrc_binary` 和 `jrc_permanent`；
* 增加 embedding 缓存，避免每次 baseline 评测都重新跑 AEF forward；
* 对 `best.pt` 与 `final.pt` 分别做下游评测，确认训练 loss 最优是否也对应下游任务最优。
